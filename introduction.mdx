The Azure DevOps Migration Tools let you bulk edit and migrate data between Team Projects on Microsoft Team Foundation Server (TFS) and Azure DevOps Services. The tools are open source, published on [GitHub](https://github.com/nkdAgility/azure-devops-migration-tools), and available as a Winget package (`nkdAgility.AzureDevOpsMigrationTools`).

<Info>
  This tool is intended for experienced users familiar with TFS/Azure DevOps object models. Not all migration scenarios are supported. Community support is available on [GitHub Discussions](https://github.com/nkdAgility/azure-devops-migration-tools/discussions).
</Info>

## Compatibility

The tools run on **Windows** and support connecting to:

- Team Foundation Server 2013 and all later versions
- Azure DevOps Server (all versions)
- Azure DevOps Services (hosted)

You can migrate from any supported source to any supported target — including across organizations, collections, and on-premises/cloud boundaries.

## What can be migrated

| Data type | Details |
|-----------|---------|
| **Work Items** | Includes history, links, attachments, and custom field mappings. Optionally includes Teams and Shared Queries. |
| **Test Plans & Suites** | Includes Configurations, Shared Steps, and Shared Parameters. |
| **Pipelines** | Build pipelines (YAML and classic) and release pipelines between Azure DevOps organizations. Requires `AzureDevOpsEndpoint`. XAML pipelines are not supported. |
| **Processes** | Process templates between locations. |

<Note>
  "Locations" refers to any combination of Projects, Collections, and Organizations.
</Note>

## Key features

- **Resumable migrations** — the tool tracks migration state using a `ReflectedWorkItemId` field on each work item, so you can stop and restart without duplicating data.
- **Work Item rules bypass** — the tool bypasses the TFS/Azure DevOps Work Item rules engine, allowing direct state transitions (e.g., setting `Closed` without going through `New`).
- **Field and type mappings** — configure field-to-field mappings and work item type mappings to handle schema differences between source and target.
- **WIQL-based scoping** — control exactly which work items are migrated using a WIQL query.
- **Revision history** — optionally replay full revision history so that the audit trail is preserved in the target.
